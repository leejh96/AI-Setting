# Backend Development Mindset (백엔드 개발자 마인드셋)

백엔드 엔지니어를 위한 문제 해결 방식, 아키텍처적 사고 및 협업 패턴 (2025).

## 문제 해결 마인드셋

### 시스템적 사고 (Systems Thinking)

**전체론적 엔지니어링:** 개별 컴포넌트가 전체 생태계 내에서 어떻게 상호작용하는지 이해합니다.

`User → Load Balancer → API → Biz Logic → Cache → DB`

**질문해야 할 것들:**
- 이 컴포넌트가 실패하면 어떤 일이 벌어지는가?
- 부하가 증가하면 어디가 병목이 되는가?
- 변경의 파급 범위(Blast radius)는 어디까지인가?

### 복잡한 문제 분해하기

1. **요구사항 이해:** 해결하려는 문제가 무엇인가?
2. **제약사항 파악:** 성능, 예산, 마감기한, 기술 스택
3. **모듈화:** 관심사 분리 (인증, 데이터, 비즈니스)
4. **인터페이스 정의:** 모듈 간 계약 설계
5. **우선순위:** 핵심 경로(Critical path) 먼저

## 트레이드오프 분석 (Trade-Off Analysis)

소프트웨어 개발은 정답이 아닌 **선택**의 문제입니다.

### CAP 이론
분산 시스템에서 **일관성(Consistency), 가용성(Availability), 분할 내성(Partition Tolerance)** 중 두 가지만 선택 가능합니다.
- **CP:** 금융 시스템 (데이터 정확성 중요)
- **AP:** 소셜 피드 (일부 데이터 지연돼도 접속 보장 중요)

### 성능 vs 유지보수성
- **성능:** 핫 경로, 실시간 시스템 (최적화된 복잡한 쿼리)
- **유지보수성:** 내부 도구, 관리자 페이지 (ORM, 읽기 쉬운 코드)
- 핵심은 **상황에 맞는 선택**입니다.

### 기술 부채 관리
부채는 피할 수 없지만 관리 가능해야 합니다. '고이자' 부채(핵심 기능의 나쁜 코드)는 즉시 갚고, '저이자' 부채는 계획적으로 미룹니다.

## 아키텍처적 사고

### 도메인 주도 설계 (DDD)
- **Bounded Context:** 각 도메인(주문, 재고, 배송)의 명확한 경계 설정.
- **Ubiquitous Language:** 개발자와 현업 전문가가 동일한 용어 사용.

### 계층형 아키텍처 (Layered Architecture)
- **Presentation:** API 엔드포인트
- **Business Logic:** 핵심 규칙
- **Data Access:** DB 처리
*각 계층은 분리되어야 하며, 변경이 서로에게 최소한의 영향을 주어야 합니다.*

### 실패를 가정한 설계 (Resilience)
"모든 것은 언젠가 실패한다"는 전제로 설계합니다.
- 서킷 브레이커, 재시도(Backoff), 타임아웃, 폴백(Fallback) 구현.

## 개발자 마인드셋

### 유지보수 가능한 코드 작성
- **SOLID 원칙** 적용
- **엣지 케이스** 고려 (Null, 빈 값, 경계값, 동시성 이슈, SQL 인젝션 등)

### 테스트 마인드셋 (TDD/BDD)
- 해피 패스(Happy path)는 AI가 짤 수 있습니다. 인간은 엣지 케이스와 실패 시나리오에 집중하세요.
- **TDD 사이클:** 실패하는 테스트 작성 -> 코드 구현 -> 리팩토링.

### 관측 가능성 (Observability)
- **"이게 왜 느리지?"** -> 메트릭 확인
- **"이게 왜 안 되지?"** -> 로그 확인
- **"어디서 멈췄지?"** -> 트레이싱 확인
디버깅하기 쉬운 코드를 작성하는 것이 고수의 특징입니다.

## 협업과 커뮤니케이션

### API 설계 = 제품 설계
API는 클라이언트 개발자가 사용하는 '제품'입니다.
- 일관성 있는 네이밍과 에러 응답
- 명확한 문서화 (Swagger)
- 하위 호환성 준수

### 코드 리뷰 마인드셋
- **예방 우선:** 보안 취약점, 성능 이슈, 에러 처리를 중점적으로 확인.
- **건설적 피드백:** "이거 틀렸어요"가 아니라 "이 방식은 SQL 인젝션 위험이 있으니 파라미터화된 쿼리를 제안합니다"라고 말하세요.
